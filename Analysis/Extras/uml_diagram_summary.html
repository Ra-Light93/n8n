Main UML Diagram Categories

1. Structural diagrams (static structure)

Used to describe system architecture, classes, components, and dependencies:

Class Diagram

Usage: Shows the static structure of a system, including classes, their attributes, methods, and the relationships between them. It's the backbone of object-oriented modeling.

Example Case: Modeling the structure of a customer order system, showing classes like `Customer`, `Order`, and `Product`, and how they relate to each other.

Components:

  • Class: Represents a blueprint for objects, typically divided into three compartments: Name, Attributes, and Operations (methods).
  • Visibility: Symbols indicating access levels for attributes and methods (e.g., public `+`, private `-`, protected `#`, package `~`).
  • Relationships: Define how classes interact:
    • Association: A general relationship between two classes.
    • Generalization (Inheritance): An "is-a" relationship where a subclass inherits from a superclass.
    • Aggregation: A "has-a" relationship where parts can exist independently of the whole.
    • Composition: A strong "has-a" relationship where parts cannot exist without the whole.
    • Dependency: One class relies on another for its functionality.
    • Realization: A class implements an interface.
  • Interfaces: A collection of operation signatures defining a cohesive set of behaviors.
  • Data Types: Classifiers defining data values (primitive types, enumerations).
  • Packages: Used to organize related classifiers within a diagram.

Package Diagram

Usage: Organizes the elements of a system into groups (packages). It helps to manage complexity by showing dependencies between packages.

Example Case: Structuring a large e-commerce application into packages like `Billing`, `Shipping`, and `UserManagement`.

Components:

  • Package: The fundamental unit, depicted as a folder-like icon, acting as a container for UML elements.
  • Packageable Element: Named elements directly owned by a package (e.g., classes, interfaces, components).
  • Dependencies: Illustrate that changes in one package might affect another:
    • Element Import: Importing a specific packageable element.
    • Package Import: Importing an entire package.
    • Package Merge: Contents of one package are extended by another.
    • Access: One package requires functionality from another.

Component Diagram

Usage: Describes how a software system is split up into components and shows the dependencies between these components.

Example Case: Showing the high-level components of a web application, such as a `WebApp`, `APIServer`, and `Database`.

Components:

  • Component: A modular, replaceable, and independent part of a system.
  • Interface: Defines operations a component either provides (lollipop notation) or requires (socket notation).
  • Port: A specific interaction point on the boundary of a component.
  • Connector: Illustrates communication and relationships between two components.
  • Dependency: A relationship indicating one component relies on another (dashed line).
  • Artifacts: Physical implementation units of a system (e.g., executable files, libraries).
  • Packages: Used to group related model elements.

Deployment Diagram

Usage: Shows the physical deployment of software artifacts on hardware nodes. It models the runtime architecture of a system.

Example Case: Illustrating how a web application's components are deployed across a web server, application server, and database server.

Components:

  • Nodes: Represent physical hardware (e.g., servers, devices) or software execution environments (e.g., OS, JVM).
  • Artifacts: Concrete, physical pieces of information deployed on nodes (e.g., executable files, libraries, configuration files).
  • Components: Modular and reusable parts of a system, implemented by artifacts.
  • Communication Paths: Associations between nodes through which they exchange signals and messages.
  • Dependencies: One node or component relies on another.
  • Interfaces: A contractual relationship a component must implement.
  • Deployment Specifications: Define deployment properties for an artifact on a node.

Object Diagram

Usage: Shows a snapshot of the instances of classes (objects) and their relationships at a specific point in time. It's a concrete representation of a class diagram.

Example Case: Displaying a specific customer's order, with object instances for that `Customer`, their `Order`, and the `Products` in that order.

Components:

  • Objects (Instance Specifications): Specific instances of classes, named as `objectName: ClassName` (underlined).
  • Attributes and Values: Attributes of an object listed with their specific values at a given time.
  • Links: Connections between objects, representing instances of associations.
  • Relationships: Various relationships between object instances (Association, Dependency, Aggregation, Composition, Generalization, Realization).

2. Behavioral diagrams (dynamic behavior)

Used to describe logic, flow, and interactions:

Use Case Diagram

Usage: Describes the functional requirements of a system from the user's perspective. It shows the interactions between users (actors) and the system to achieve a goal.

Example Case: Modeling how a `Customer` can `ViewProducts`, `AddToCart`, and `Checkout` in an online store.

Components:

  • Actors: External entities (users, systems) interacting with the system, typically stick figures.
  • Use Cases: Specific functions or actions the system performs, represented as horizontal ellipses.
  • System Boundary: A rectangle defining the scope of the system.
  • Relationships:
    • Association: Interaction between an actor and a use case.
    • Include: One use case incorporates another's functionality.
    • Extend: One use case adds optional functionality to another.
    • Generalization: "Is-a" connection between use cases or actors.

Sequence Diagram

Usage: Shows how objects interact with each other over time. It emphasizes the time-ordered sequence of messages exchanged between objects.

Example Case: Detailing the sequence of events when a user logs in, from the UI to the authentication service and the database.

Components:

  • Lifeline: Represents an individual participant or object, with a vertical dashed line indicating its existence over time.
  • Message: Depicts communication between lifelines (Synchronous, Asynchronous, Self-Message, Return Message).
  • Execution Specification (Activation Bar): A thin rectangle showing when an object is actively performing an operation.
  • Combined Fragment: Used to model complex interactions (e.g., `alt` for alternative, `opt` for optional, `loop` for repetition, `par` for parallel).
  • Interaction Use: Allows for reusing a predefined sequence of interactions.
  • State Invariant: A condition that must hold true for a lifeline at a specific point.
  • Destruction Occurrence: An 'X' mark indicating an object is destroyed.
  • Gate: A connection point for messages entering or exiting a combined fragment.

Activity Diagram

Usage: Models the flow of control from one activity to another. It's useful for describing business processes and workflows.

Example Case: Outlining the steps of an order fulfillment process, from receiving the order to shipping the product.

Components:

  • Initial Node: Marks the starting point of the activity flow (solid black circle).
  • Activity/Action State: Represents a single step or task (rounded-corner rectangle).
  • Control Flow: Arrows indicating the direction and sequence of activities.
  • Decision Node: A diamond shape where the flow branches based on a condition.
  • Merge Node: A diamond shape that brings together alternative flows.
  • Fork Node: A thick line to split a single flow into multiple concurrent activities.
  • Join Node: A thick line that synchronizes and combines multiple concurrent activities.
  • Final Node: Indicates the end of the activity flow (Activity Final, Flow Final).
  • Object/Object Flow: Represents the movement of data or objects.
  • Activity Partition (Swimlane): Rectangular areas to group related activities and assign responsibility.
  • Guard: A condition on a control flow that must be true to proceed.

State Machine Diagram

Usage: Describes the different states an object can be in and how it transitions from one state to another in response to events.

Example Case: Modeling the states of an online order, such as `Pending`, `Processing`, `Shipped`, and `Delivered`.

Components:

  • States: Conditions an object can exist in (rounded rectangles: Simple, Composite, Initial, Final).
  • Transitions: Directed arrows showing paths between states, triggered by events, with optional guards and effects.
  • Events: Occurrences that trigger transitions.
  • Actions: Operations executed in response to events or state changes (Entry, Exit).
  • Pseudostates: Abstract vertices structuring transitions (e.g., Choice, Junction, Fork, Join, History States).
  • Regions (Orthogonal Regions): Used within composite states to model concurrent behavior.

Communication Diagram

Usage: Shows the interactions between objects in terms of the messages exchanged. It emphasizes the relationships between objects rather than the time sequence.

Example Case: Illustrating how the objects in a "place order" scenario collaborate, focusing on the links between them.

Components:

  • Objects/Lifelines: Active participants in the interaction (rectangles).
  • Actors: External entities interacting with the system (stick figures).
  • Links: Lines connecting objects, indicating an association.
  • Messages: Arrows on links showing direction of communication, with name and sequence number.
  • Sequence Numbers: Numerical prefixes defining chronological order of messages.
  • Interaction Occurrences: Specific situations where interactions take place.
  • Roles: Define the specific part an object plays.
  • Interaction Frames: Provide a contextual boundary for the diagram.
